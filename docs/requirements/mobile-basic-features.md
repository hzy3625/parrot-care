# Android 移动端基础功能设计

## 1. 系统目标

移动端的目标不是做医疗诊断，而是帮助鹦鹉饲养者在手机上完成本地化、可追溯的健康与声音记录：

```text
建立鹦鹉档案
  -> 录制 / 导入声音 / 记录无音频观察
  -> 标记记录用途：日常基线或异常观察
  -> 人工选择类别、状态和备注
  -> 保存到本机
  -> 按鹦鹉、类别、状态、时间查看和复盘
  -> 导出备份，在另一台手机导入
```

本阶段优先完成“记录和管理”的基本能力，为后续 ML 训练和自动识别积累高质量本地数据。

## 2. 使用场景

### 2.1 日常基线记录

用户定期录制鹦鹉正常状态下的叫声，作为后续对比素材。

典型动作：

- 选择鹦鹉。
- 点击录音。
- 试听后保存为“日常基线”。
- 类别选择“正常鸣叫”或“待确认录音”。
- 填写备注，例如“早上进食后正常叫声”。

### 2.2 异常声音记录

用户发现持续尖叫、夜间惊飞、异常安静等情况时，快速记录声音。

典型动作：

- 选择鹦鹉。
- 录制或导入已有音频。
- 保存为“异常观察”。
- 人工选择类别和状态。
- 后续在详情页更新为“观察中”“已处理”或“已排除”。

### 2.3 无音频健康观察

部分场景没有音频，但仍需要记录，例如啄羽、精神差、食欲变化。

典型动作：

- 选择鹦鹉。
- 新建“观察记录”。
- 类别选择“啄羽行为”“异常安静”或“待确认录音”。
- 填写备注。
- 后续修改状态和备注。

### 2.4 多只鹦鹉管理

一个用户可能同时饲养多只鹦鹉。所有记录必须归属到具体鹦鹉，避免混淆。

典型动作：

- 创建多个鹦鹉档案。
- 录制时选择当前鹦鹉。
- 记录列表按鹦鹉筛选。

### 2.5 换机和数据备份

移动端核心数据存储在本机。用户换手机时，需要通过文件完成迁移。

典型动作：

- 在旧手机导出 ZIP 备份。
- 在新手机导入 ZIP。
- 导入后保留鹦鹉档案、事件记录、音频文件、类别、状态和备注。

## 3. 范围与边界

### 3.1 本阶段包含

- 原生 Android Java/XML。
- 本地鹦鹉档案。
- 本地录音。
- 本地音频导入。
- 无音频观察记录。
- 人工类别、用途、状态和备注。
- 本地列表、详情、播放、修改和删除。
- 本地 ZIP 备份导出和导入。
- 离线可用。

### 3.2 本阶段不包含

- ML 训练、端侧模型和自动声音分类。
- 服务端分析、账号登录和云同步。
- 后台持续监听、定时任务和异常通知。
- 医疗诊断或自动健康结论。
- iOS、WebView 或跨平台运行时。

所有核心功能必须在无网络、未启动 API、未登录账号的情况下工作。

## 4. 当前实现基线

当前 Android 版本已经具备：

- 原生 Java/XML 工程。
- 运行时申请麦克风权限。
- 使用 `MediaRecorder` 录制 AAC/M4A 音频。
- 音频保存到应用私有目录。
- 使用 `SQLiteOpenHelper` 写入一条基础事件记录。
- 离开页面时取消未完成录音。
- 无第三方运行时依赖，可通过 Android SDK 命令行工具构建。

当前缺口：

- 没有鹦鹉档案。
- 记录不能归属到具体鹦鹉。
- 看不到已保存记录。
- 不能试听、重新录制或导入已有音频。
- 不能创建无音频观察。
- 不能人工分类、选择记录用途、填写备注或修改状态。
- 不能查看详情、修改和删除记录。
- 没有本地数据备份与恢复能力。

## 5. 功能优先级

### P0：本地记录闭环

P0 完成后，移动端才算具备基本可用能力。

#### 5.1 首页

首页按操作频率组织，不做复杂导航：

1. 高频入口：记录
   - 录制声音。
   - 导入音频。
   - 新建观察。
2. 常用入口：记录库
   - 查看全部记录。
   - 最近 5 条记录。
3. 低频入口：数据和设置
   - 鹦鹉档案管理。
   - 导出备份。
   - 导入备份。

要求：

- 最近记录按创建时间倒序。
- 首次使用显示简短说明，不自动创建示例记录。
- 无鹦鹉档案时，引导用户先创建档案；也允许自动创建一个“默认鹦鹉”，降低首次使用门槛。

#### 5.2 鹦鹉档案

最小字段：

- 名称，必填。
- 品种，可选。
- 出生日期或年龄，可选。
- 备注，可选。

能力：

- 新建鹦鹉档案。
- 修改档案。
- 切换当前鹦鹉。
- 记录列表按鹦鹉筛选。

删除档案本阶段不作为 P0 必需能力，避免误删关联记录。可以先只提供“修改”。

#### 5.3 录音

- 点击录音时才申请麦克风权限。
- 显示录音时长和正在录音状态。
- 支持停止、取消和重新录制。
- 录音停止后进入预览，不立即写入正式记录。
- 预览阶段支持播放、暂停和重新录制。
- 过短、空文件或录制失败时不得创建数据库记录。
- 应用进入后台时停止并丢弃尚未确认的录音，避免隐私和状态问题。

#### 5.4 导入音频

- 使用系统文件选择器选择音频。
- 支持常见系统可解码格式，至少覆盖 M4A、MP3 和 WAV。
- 将选中文件复制到应用私有目录，不长期依赖外部 URI 权限。
- 导入后进入与录音相同的预览和保存流程。
- 非音频、空文件或无法播放的文件给出可操作错误提示。

#### 5.5 新建观察

无音频观察用于记录行为、状态和护理动作。

- 不申请麦克风权限。
- 不要求音频文件。
- 必须选择鹦鹉。
- 必须选择类别。
- 默认用途为“异常观察”，允许改为“日常基线”。
- 支持填写备注。
- 保存后进入记录详情页。

#### 5.6 保存记录

保存前必须确认：

- 归属鹦鹉。
- 记录用途。
- 人工类别。
- 状态。
- 备注，可为空。

记录用途：

- `baseline`：日常基线。
- `abnormal`：异常观察。
- `manual`：无音频观察。

状态：

- `pending`：待处理。
- `reviewing`：观察中。
- `resolved`：已处理。
- `false_positive`：已排除。

人工类别与 Web 端保持一致：

- 正常鸣叫 `normal_chirp`
- 持续尖叫 `scream`
- 夜间惊飞 `night_fright`
- 啄羽行为 `plucking`
- 异常安静 `silence`
- 待确认录音 `unclassified`

保存成功后进入记录详情页。

数据库记录和音频文件必须保持一致：任一步失败时清理未完成数据。

#### 5.7 记录列表

- 默认按创建时间倒序排列。
- 每行显示鹦鹉、类别、用途、状态、时间和时长。
- 无音频观察显示“无音频”。
- 提供鹦鹉、类别、用途和状态筛选。
- 点击记录进入详情。
- 列表保持紧凑，不直接加载完整音频内容。
- 从详情返回后保留筛选条件和滚动位置。

#### 5.8 记录详情

- 显示鹦鹉、类别、用途、状态、创建时间、时长和备注。
- 有音频时支持播放、暂停和从头播放。
- 无音频时不显示播放按钮。
- 支持修改类别。
- 支持修改用途。
- 支持修改状态。
- 支持修改备注。
- 支持删除记录，删除前二次确认。
- 删除音频记录时同步删除应用私有目录中的音频文件。
- 音频文件缺失或损坏时，详情仍可打开并允许删除记录。

### P1：本地数据安全

P1 建议在首个可外部试用版本中完成。

#### 5.9 数据导出

- 导出一个带版本号的 ZIP 备份文件。
- 备份包含鹦鹉档案、事件元数据清单和对应音频文件。
- 通过系统分享或文件保存面板交给用户。
- 不包含账号凭据、设备标识、API Token 或未来的云同步凭据。

#### 5.10 数据导入

- 使用系统文件选择器选择备份 ZIP。
- 导入前校验备份版本、清单格式和音频文件。
- 使用新 UUID 处理 ID 冲突，不覆盖现有记录。
- 如果备份事件引用的鹦鹉不存在，导入对应鹦鹉档案；仍异常时归到“默认鹦鹉”。
- 单个错误条目不应破坏已经存在的本地数据。
- 完成后展示成功、跳过和失败数量。

### P2：后续增强

P2 不进入当前开发范围，但数据结构要给后续留扩展点。

- 简单趋势视图：按天/周展示异常记录数量。
- 日常基线对比：同一鹦鹉的正常声音和异常声音人工对照。
- ML 自动分类：使用后续训练模型或服务端分析结果填充建议类别。
- 云同步和多设备协同。

## 6. 页面结构

保持简单 Activity，不引入复杂导航框架：

```text
首页 MainActivity
  |- 录制声音 -> RecordActivity
  |- 导入音频 -> Import/Record preview flow
  |- 新建观察 -> ObservationActivity
  |- 最近记录 -> DetailActivity
  |- 查看全部 -> LibraryActivity
  |- 鹦鹉档案 -> ParrotActivity
  `- 数据管理 -> DataActivity

记录列表 LibraryActivity
  `- 记录详情 DetailActivity

数据管理 DataActivity
  |- 导出备份
  `- 导入备份
```

使用 Android 平台 `Activity`、XML Layout 和显式 `Intent`。不引入 AndroidX Navigation、Compose 或 Fragment 框架。

## 7. 本地数据模型

建议将数据库升级为版本 2 或更高，并对现有表做前向迁移。

### 7.1 鹦鹉表

```sql
CREATE TABLE parrots (
    id TEXT PRIMARY KEY,
    name TEXT NOT NULL,
    species TEXT NOT NULL DEFAULT '',
    birthday TEXT NOT NULL DEFAULT '',
    note TEXT NOT NULL DEFAULT '',
    created_at INTEGER NOT NULL,
    updated_at INTEGER NOT NULL
);
```

### 7.2 事件表

```sql
CREATE TABLE events (
    id TEXT PRIMARY KEY,
    parrot_id TEXT NOT NULL,
    category_id TEXT NOT NULL,
    status TEXT NOT NULL,
    purpose TEXT NOT NULL,
    audio_path TEXT,
    source TEXT NOT NULL,
    duration_ms INTEGER NOT NULL DEFAULT 0,
    note TEXT NOT NULL DEFAULT '',
    created_at INTEGER NOT NULL,
    updated_at INTEGER NOT NULL
);
```

字段约束：

- `purpose` 只允许 `baseline`、`abnormal`、`manual`。
- `source` 只允许 `recording`、`import`、`manual`。
- `source=manual` 时 `audio_path` 可以为空，`duration_ms=0`。
- `source=recording` 或 `source=import` 时 `audio_path` 必须有值。
- 音频文件只放在 `files/audio/`。
- 数据库只保存稳定的应用内相对路径，不保存音频二进制。
- 当前不保存 `confidence`、模型版本或自动风险等级。
- 迁移必须保留已经录制的旧数据；旧数据没有 `parrot_id` 时归到“默认鹦鹉”。

## 8. Java 模块拆分

```text
com.parrotcare.mobile/
  MainActivity.java
  parrot/ParrotActivity.java
  record/RecordActivity.java
  observation/ObservationActivity.java
  library/LibraryActivity.java
  detail/DetailActivity.java
  data/DataActivity.java
  audio/AudioRecorder.java
  audio/AudioPlayer.java
  data/LocalStore.java
  data/AudioFileStore.java
  data/BackupStore.java
  model/ParrotProfile.java
  model/ParrotEvent.java
  model/EventCategory.java
  model/EventPurpose.java
  model/EventStatus.java
```

职责边界：

- Activity 负责页面状态、系统权限和用户操作。
- `AudioRecorder`、`AudioPlayer` 只处理音频生命周期。
- `LocalStore` 负责 SQLite 查询和迁移。
- `AudioFileStore` 负责复制、删除和校验音频文件。
- `BackupStore` 负责 ZIP 清单和文件导入导出。
- 数据写入、文件删除和失败回滚由用例方法协调，不能散落在按钮监听器中。

## 9. 开发顺序

1. 扩展数据模型：新增鹦鹉档案、用途字段、无音频事件能力和迁移逻辑。
2. 实现鹦鹉档案管理和首页当前鹦鹉选择。
3. 实现 `AudioPlayer`，完成录音后的预览、重新录制和保存。
4. 实现记录保存表单：鹦鹉、用途、类别、状态、备注。
5. 实现记录列表和详情，打通播放、编辑、删除闭环。
6. 实现无音频观察记录。
7. 实现系统文件选择和音频导入。
8. 完善权限拒绝、后台切换、坏文件和空间不足等错误处理。
9. 完成备份导出与导入。
10. 构建版本化 APK，并执行真机回归。

在第 1 至 8 步完成并通过验收前，不接入 ML 或 API。

## 11. 完成标准

代码开发完成必须同时满足：

- `make check-android check-structure` 通过。
- `make android-release` 生成版本化 APK。
- APK 不包含第三方运行时、模型、密钥或用户数据。
- 在至少一台 Android 真机完成 P0 全流程测试。
- 验证拒绝权限、离线启动、进程重启和升级保留数据。
- 更新 `apps/mobile/release/android/README.md` 的版本、构建与测试记录。
